## Section 2 — Landscape Map

### 1. 主流路线：DPU/SmartNIC 内置硬件引擎

NVIDIA BlueField-2/3 的 C-engine（DEFLATE 系）是目前研究最系统的商用平台：Li et al. 构建了从 benchmark（Cluster 2023）→ 瓶颈分析（Micro 2024）→ PEDAL 调度库（IPDPS 2024）的完整研究链。核心结论是硬件引擎本身速度够用，但 DMA 调度开销和 API 抽象层是真正瓶颈。这一方向研究已相对成熟，进一步创新需要切入 RDMA 协议交互层面。

### 2. FPGA 参考实现：LZ4 加速器生态 + RoCE BALBOA 突破

FPGA 上的 LZ4/Snappy 加速器形成独立分支：从 FPL 2019 的首批完整实现，到 2024 年 arXiv 的 16.10 Gb/s 高吞吐设计，再到 CAST 商用 IP Core。**RoCE BALBOA**（ETH Zurich，EuroSys 2025）将这一分支与 RDMA 直接结合，是本次新发现的最重要论文：它提供了开源 100G FPGA RoCE v2 栈，并原生支持 payload compression（同时含 DPI 和 AES），将压缩直接嵌入 RDMA 数据路径。这标志着"在 FPGA 上同时运行 RDMA+compression"从概念进入开源可验证阶段。RoCE BALBOA 还是实验平台的候选——已在多 FPGA 集群上部署运行。

### 3. 算法/硬件协同设计（NetZIP 代表方向）

MICRO 2025 的 NetZIP 是目前最系统的 in-network compression 工作：bump-in-the-wire 架构 + 定制 LZ4 变体（byte/bit grouping + delta）+ 面向 FPGA 的硬件设计。分布式训练场景实测 35% 训练时间缩短。它的局限：专注于分布式训练（AllReduce），未涉及通用 RDMA 数据路径、拥塞控制交互、以及 Rx 解压带宽膨胀问题。

### 4. 跨域：内存/存储压缩的可迁移技术

BPC/EBPC（ISCA 2016 / TNNLS 2019）以及 **Ecco**（ISCA 2025）代表内存侧压缩前沿。Ecco 的熵感知 Huffman + 并行多级解码流水线，将 GPU 缓存解码延迟降低两个数量级——这个设计可直接迁移到 NIC Rx 解压路径（面临同样的高速数据流 + 低延迟解码约束）。**ASIC 压缩加速器部署策略分析**（arXiv 2025）系统对比了 Zipline/QAT/ScaleFlux 三种架构策略，对 NIC 内置 vs 外挂 vs chiplet 的设计空间有直接参考价值。

新补录的 **ZipServ**（arXiv 2026-03-18）代表了 GPU 推理侧的最新进展：它提出了**无变长 bitstream 的并行解码编码方案**，解决了传统 Huffman/LZ 压缩在 GPU 上并行解码困难的问题（变长编码导致数据依赖，无法并行展开）。这个设计思路与 NIC Rx 解压路径的需求高度对称——NIC 硬件同样需要在高速数据流下做并行解压，且变长 bitstream 在 FPGA 流水线中同样是瓶颈。ZipServ 的编码方案可作为 NIC 侧算法设计的直接参考。

## Section 3 — Structural Gaps

### 1. Cross-domain transfer
- **Ecco（ISCA 2025）的并行 Huffman 解码流水线**用于 GPU 内存缓存，未被应用到 RDMA NIC Rx 解压路径。NIC Rx 有完全对称的问题：解压 payload 膨胀可超过 PCIe 带宽上限（400Gbps NIC 以 3:1 压缩率接收时，解压后需 ~1.2Tbps）。
- **ZipServ（arXiv 2026）的"无变长 bitstream 并行解码"编码方案**解决了 GPU 上并行解压的数据依赖问题，但未被迁移到 FPGA NIC 上。FPGA 流水线同样受变长 bitstream 制约，ZipServ 的定长 block 编码设计是直接可迁移的解决思路。
- **BPC/EBPC 的 DBX 变换**专为 GPU 片外内存设计，但 AllReduce gradient 数据同样流经 RDMA NIC，数据统计特性相同。尚无人将 BPC 嵌入 RDMA 数据路径做 bump-in-the-wire 压缩。

### 2. Contradictory findings
- NetZIP 声称压缩"不需要端主机配合"，RoCE BALBOA 实现显示 payload size 变化需要修改 RDMA Verb/WQE 头部（compression flags 字段）。两者对"是否需要修改 RDMA 协议语义"立场存在分歧，是悬而未决的设计决策。

### 3. Untested assumptions
- **"压缩率稳定，不影响流量控制"**：所有现有工作均假设压缩率变化不大，ECN/PFC 阈值可静态配置。但 ML 数据的压缩率在不同 layer、不同 step 间可从 1.1x 到 4x 波动，导致 DCQCN 速率反馈产生错误估计——**这个假设从未被系统验证**。
- **"解压延迟对 RDMA RTT 无显著影响"**：现有论文报告压缩吞吐，但无人分析 tail latency（P99/P999 解压时间）如何影响 one-sided RDMA READ 完成时间。

### 4. Unexplored regimes
- **Rx 侧解压带宽膨胀**：400Gbps NIC 以 3:1 压缩率接收时，解压后需约 1.2 Tbps PCIe/内存带宽，超过 PCIe Gen5 x16 上限（~256 GB/s）。DPU 上问题更严重。NetZIP 和 RoCE BALBOA 均未系统分析此问题。
- **多租户场景下压缩优先级与隔离**：多个 tenant 共享 NIC 压缩引擎时的调度策略（参考 COER 在 CC 中的协同设计思路）尚无研究。

### 5. Unasked diagnostic questions
- **真实 RDMA 工作负载的压缩率分布特征**：大模型训练各 layer gradient/activation 的 byte entropy 分布？不同模型结构（Transformer vs MoE）间的差异？这是设计 NIC 侧自适应压缩算法的必要前提，缺乏系统 measurement study。
- **RoCE BALBOA compression 模块对 QoS/隔离的影响**：开源 RoCE BALBOA 已集成 payload compression，但尚未有论文分析在 FPGA 上压缩与 RDMA 协议栈共享资源时，P99 延迟和吞吐公平性的影响。

## Section 4 — Competitive Landscape

| 竞品 | 声称解决 | 留下的空白 | 类型 | 与我们的关系 |
|------|----------|------------|------|--------------|
| **NetZIP** (MICRO 2025) | bump-in-the-wire in-network 无损压缩，降低分布式训练通信开销，算法/硬件协同 | 未处理 RDMA 协议交互（MTU、CC 感知）；仅针对 AllReduce；无 Rx 带宽膨胀分析 | 同行评审 (MICRO) | 直接竞品；需明确差异化 |
| **RoCE BALBOA** (EuroSys poster/arXiv 2025) | 开源 FPGA RDMA 栈 + 原生 payload compression + DPI + AES | 仍是 poster/preprint；compression 集成深度/实验数据有限；无 CC 交互分析；开放设计空间未封闭 | preprint + poster | 潜在平台基础 + 部分竞品 |
| **PEDAL** (IPDPS 2024) | BlueField DPU C-engine + SoC 协同调度，集成 MPICH | 绑定 BlueField 商用平台；DPU 专有 API 不通用；无协议层修改 | 同行评审 (IPDPS) | 间接竞品；平台不同 |
