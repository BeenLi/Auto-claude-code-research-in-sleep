## 领域全景（更新版）

### 1. 主流路线：基于 LZ4/DEFLATE 的 NIC 硬件加速

NIC/DPU 侧无损压缩已从概念走向实测。NVIDIA BlueField-2/3 上的 C-engine（基于 DEFLATE）是目前研究最充分的平台，2023-2024 年有一条完整研究链：benchmark（Cluster 2023）→ 瓶颈分析（Micro 2024）→ PEDAL 加速库（IPDPS 2024）。核心结论：**硬件 C-engine 相比软件有 26x+ 加速，但 DMA 调度开销是主要瓶颈**，需要软件库层面协同优化。

LZ4 是 FPGA 实现的主流选择，兼顾压缩率与硬件友好性。CAST 在 2024 年发布首款 LZ4+Snappy 双模 ASIC/FPGA IP Core，标志着产品化进入。inline LZ4 on Corundum FPGA SmartNIC 实际案例显示，日志类负载下有效带宽可提升 70%。

### 2. 🆕 最新进展：In-Network 压缩系统化（NetZIP, MICRO 2025）

**NetZIP** 是本次新发现的最重要论文，代表了 in-network 压缩的最新系统化工作：
- 采用 **bump-in-the-wire** 设计，压缩/解压透明插入网络路径，不需要端主机配合
- 对 LZ4 算法本身做了 **co-design**：byte grouping（相似字节集中）+ bit-plane grouping + delta 编码，让 FPGA 硬件实现更高效
- 在分布式训练场景实测显示 **35% 训练时间减少**，直接验证了 in-network 压缩对 ML 集合通信的价值

与 BlueField C-engine 系列不同，NetZIP 不依赖特定 DPU 产品，更接近"通用 FPGA SmartNIC 上的参考实现"。这是当前文献中与我们潜在工作最接近的竞品。

### 3. BPC/EBPC 路线：面向 ML 张量数据的专用压缩

BPC（ISCA 2016）和 EBPC（IEEE TNNLS 2019）代表了另一个维度——**针对 ML 数据类型（整数/浮点张量）的专用压缩算法**：
- DBX（Delta+Bit-Xor）变换利用张量数值在相邻元素间高度相关的特性，比通用 LZ4 对 ML 数据有更好的压缩率（4-5x vs ~2x）
- EBPC 已有 65nm ASIC 实现，面积开销小，适合集成到 NIC/DPU
- 但两篇论文的目标场景是 **片外内存带宽**（DRAM BW 节省），而非网络传输；如何将 BPC 思路迁移到 RDMA 数据路径是一个开放研究问题

### 4. RDMA 集成方向的空白

现有工作在以下方面仍有明显空白（具有研究价值）：
1. **RDMA/RoCE 数据路径内嵌压缩**：压缩与 RDMA 协议栈的协同（压缩改变 payload 大小，影响 MTU 对齐、PFC/ECN、Credit 流控）
2. **压缩 aware 拥塞控制**：压缩率动态变化导致带宽利用率波动，现有 ECN/DCQCN 未感知这一变化
3. **解压延迟对 RDMA RTT 的影响**：硬件解压路径的 tail latency 如何影响 one-sided RDMA 操作的完成时间，现无系统性分析
4. **BPC/EBPC 在 NIC 数据路径的实现**：将 ML 专用压缩算法嵌入 SmartNIC，针对 AllReduce gradient/activation 数据做 bump-in-the-wire 压缩

### 5. 周边相关工作（低相关性）

- **OptiNIC** (arXiv 2512.22743, 2025)：通过"允许 ML 容忍部分丢包"来简化 RDMA 传输层，实现 2x 训练提速、3.5x P99 延迟降低。与压缩方向不同，但揭示了一个互补思路：传输层与 ML 语义的协同优化。
