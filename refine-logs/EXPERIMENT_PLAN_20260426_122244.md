# Experiment Plan

**Idea**: Rx Expansion Budgeting for Compressed RDMA
**Generated**: 2026-04-26 10:26 CST
**Oracle-pro browser review update**: 2026-04-26 12:22 CST
**Effort**: beast
**Validation status**: Not started

## Core Claims

1. Under sufficient offered load or burst compression, compressed RDMA can shift the bottleneck from link bytes to decompressed output service, and naive wire-byte admission can create avoidable Rx stalls/tail latency.
2. Short-window expansion-ratio estimation predicts receiver pressure accurately enough for scheduling and feedback.
3. Expansion-aware Rx budgeting improves p99/p999 FCT, completion latency, drop/stall bytes, and fairness versus naive compressed RDMA.
4. The mechanism is practical for NIC/DPU hardware: bounded metadata, SRAM, and scheduling latency.

## Oracle-Pro Review Corrections Before Implementation

The oracle-pro browser review scored the plan **5/10, verdict: almost, confidence: high**. The idea remains viable, but implementation must start by tightening the contract and baselines rather than jumping straight into htsim/gem5.

Pre-implementation fixes:

- Write a compressed-RDMA contract: compression granularity, metadata fields, original length vs compressed length, address advancement, chunk boundaries, retransmission unit, ICRC/checksum domain, ACK/completion point, and incompressible-chunk behavior.
- Separate two offered-load modes: same original-byte workload and compressed-line-rate saturated workload.
- Use a ratio-trace input file instead of hard-coded ratio sweeps only.
- Add the strongest non-oracle baseline: output-rate throttled compressed RDMA via a receiver decompressed-byte token bucket.
- Explicitly bound Workflow 1.5 claims. Allowed: analytical feasibility, queueing benefit, sensitivity, metadata/SRAM estimate. Disallowed without hardware: line-rate RoCE correctness, PCIe transaction-level behavior, timing closure, energy, production DCQCN/PFC behavior, and end-to-end LLM serving speedup.

## Baselines

- Uncompressed RDMA.
- Naive compressed RDMA: compress/decompress without output-byte control.
- Throttled compressed RDMA: receiver decompressed-byte token bucket, no EARB estimator/scheduler.
- Exact-original-byte receiver credits: uses exact original-byte length metadata when available.
- Static per-QP decompressed-byte cap.
- Static partitioning of Rx output bandwidth.
- FIFO shared decompression queue.
- Weighted fair sharing by wire bytes.
- Reactive QoS controller using only queue occupancy.
- Oracle output-byte scheduler (upper bound).

## Workloads

- Synthetic LLM collective flows with ratio classes: 1.0x, 1.2x, 1.5x, 2.0x, 3.0x, 4.0x.
- Ratio-trace files with per-chunk compressed bytes, original bytes, timestamp/phase, flow/QP/tenant, message size, and codec family.
- Concrete tensor-inspired sources: NetZIP-style gradient/activation classes if artifacts are available, public or generated FP8/e4m3 tensor dumps, measured-entropy synthetic tensors, incompressible control packets, and mixed metadata/payload traffic.
- Tensor-inspired phase traces: FFN/attention/embedding-style bursts, e4m3-like distributions, activation/gradient-like bursts, KV/checkpoint-like bursts.
- Multi-tenant mixes: latency-critical inference traffic plus elastic training/checkpoint bursts.
- Stress regimes: incast, mixed ratios, bursty arrivals, slow host-output path, limited Rx SRAM.
- Negative regimes: low compression, high PCIe bandwidth, large Rx SRAM, slow wire link, and uniform ratios.

## Metrics

- Network/RDMA: FCT p50/p95/p99/p999, retransmitted bytes, ECN/PFC marks, queue occupancy.
- Receiver pressure: decompressed output_write_gbps, Rx SRAM occupancy, expansion debt, decompression engine utilization, stall/drop bytes.
- System-facing: completion time, TTFT/TPOT proxy for serving traces, fairness/SLO-goodput.
- Compression: ratio distribution, accepted/dropped compressed bytes, expansion prediction error.
- Hardware cost: per-flow state bytes, scheduler cycles, SRAM footprint, estimated LUT/BRAM/DSP for optional RTL sketch.
- Early deterministic metrics: max expansion debt, output queue occupancy, decompressor busy time, dropped/stalled compressed bytes, and completion delay.

## Milestone 0 - Compressed-RDMA Contract

- Define supported RDMA operations and where compression is inserted.
- Define metadata fields: compressed length, original length, chunk ID, flow/QP/tenant ID, incompressible flag, checksum/ICRC handling.
- Define address advancement, packet/chunk boundaries, retransmission unit, ACK/completion point, and failure behavior.
- Define the first feedback hook to model: receiver decompressed-byte token bucket first; DCQCN/PFC/CNP feedback only after the local receiver model is validated.
- **Gate**: htsim/standalone simulator has an unambiguous byte-domain and completion contract before any performance number is claimed.

## Milestone 1 - Analytical Break-Even Model

- Build a byte-domain model: wire_bw, compression_ratio, decompressor_bw, PCIe_bw, host_write_bw, Rx_sram.
- Derive safe-admission condition: `wire_bw * expansion_ratio <= min(decompressor_bw, output_bw_budget)`.
- Report break-even surfaces for **ratio threshold**, not only fixed ratio sweeps.
- Compare same original-byte workload vs compressed-line-rate saturated workload.
- Model receiver output paths separately: NIC-to-host DRAM, NIC-to-GPU-memory-like DMA, and NIC-to-staging/CXL-like memory.
- Sweep 100/200/400/800Gbps links and PCIe 4/5/6 assumptions.
- **Gate**: show a non-contrived percentile of real or tensor-derived ratio traces where compressed-wire utilization is safe while output-service utilization exceeds 1.0 under PCIe/host/GPU-direct budgets.

## Milestone 2 - htsim Compression-Ratio Injection

- Add per-flow compressed/original byte accounting to the htsim workload adapter.
- Model decompression as both input-byte work and output-byte work.
- Model Rx output buffer/SRAM as a finite chunk-level resource.
- Define exactly which RDMA/control signals are modeled: queue occupancy, ECN/CNP, PFC pauses, retransmission, receiver credits, ACK/completion delay, or none. Omitted signals are excluded from claims.
- Compare uncompressed, naive compressed, static cap, FIFO, and EARB.
- **Gate**: EARB reduces p99 FCT or stall bytes by >=20% versus the strongest non-oracle baseline, including receiver decompressed-byte token bucket, in at least two realistic mixed-ratio regimes without severe p50/mean regression.

## Milestone 3 - gem5/htsim Window Co-Simulation

- Use window-level co-simulation: htsim emits ingress/compressed bursts; gem5 approximates host-output write/cache pressure; feedback adjusts Rx budget.
- Track PCIe/host-memory bottleneck transitions.
- Treat this as calibrated sensitivity unless host DMA/write/cache behavior is validated. Do not claim production PCIe/DDIO/RNIC behavior from this milestone alone.
- **Gate**: closed-loop simulation reproduces the hidden-bottleneck result from Milestone 1 and validates EARB feedback under clearly stated host-output assumptions.

## Milestone 4 - Ablations

- No expansion estimator (wire-byte only).
- Perfect expansion oracle.
- Stale expansion estimator.
- EWMA over previous chunks.
- Stale per-layer estimate.
- Tenant-level estimate.
- Adversarial phase change.
- No per-tenant fairness.
- Small/medium/large Rx SRAM.
- Fixed vs adaptive decompression-engine scheduling.
- Output-token-bucket baseline vs full EARB scheduler.

## Milestone 5 - Hardware Feasibility Sketch

- Estimate metadata: per-flow ratio EWMA, output-credit counter, queue pointer, tenant class.
- Sketch scheduler critical path and SRAM table size for 1K/8K/64K flows.
- Define metadata bits per flow, fixed-point ratio width, update frequency, SRAM banking/ports, output-buffer granularity, and decompressor/DMA-writer interface.
- Optional HLS/RTL micro-sketch for output-credit scheduler only; codec RTL is out of scope unless Idea 2 is selected.
- **Gate**: metadata and scheduler latency plausibly fit NIC pipeline sideband/control path.

## Stop/Kill Criteria

- If realistic ratio distributions never create output-byte pressure at 400/800Gbps, pivot to Idea 2 codec design or Idea 4 DPU QoS.
- If EARB does not beat receiver decompressed-byte token bucket, reframe or abandon; do not claim a new control mechanism.
- If htsim/gem5 integration dominates effort, first publish a standalone simulator result and keep co-sim as follow-up.
- If EARB only helps under contrived ratios, reframe as a negative measurement paper or abandon.
